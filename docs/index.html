Patient Management System

A backend-focused healthcare platform built as a distributed system of 6 independent Spring Boot microservices. Each service owns its domain, its schema, and its failure boundary. Services communicate via synchronous REST for reads and asynchronous Kafka events for writes that cross domain lines.

Spring Boot 3.4 Spring Cloud Gateway PostgreSQL 15 Apache Kafka (KRaft) Docker Compose Resilience4j Prometheus + Grafana gRPC / Protobuf
services 6 + 1 gateway
database PostgreSQL (shared instance, isolated schemas)
messaging Kafka KRaft — no ZooKeeper
test coverage 72% instruction (patient-service)
01

System Architecture

All inbound traffic enters through a single reactive API Gateway running on port 4004. The gateway handles JWT validation before requests reach any downstream service — downstream services themselves have no auth middleware. Inter-service calls that require entity verification (e.g. appointment creation verifying patient and doctor existence) are synchronous REST calls guarded by Resilience4j circuit breakers.

Database Isolation

All services connect to the same PostgreSQL instance but operate on isolated tables. Hibernate DDL is set to update — schema migrations are applied automatically on service startup. No cross-service joins; each service owns its own data model.

02

Service Data Models

Each service exposes a REST API and persists its own entity graph. Below are the primary domain models per service.

patient-service :8080

Source of truth for patient identity. Publishes PATIENT_CREATED and PATIENT_DELETED events to Kafka asynchronously via CompletableFuture.

id UUID // PK, auto-generated
name String // max 100 chars
email String // unique constraint
address String
dateOfBirth LocalDate
registeredDate LocalDate
doctor-service :8083

Manages doctor profiles and availability. Exposes a gRPC server for low-latency internal queries. Uses @Version optimistic locking on the Doctor entity.

id UUID
specialization ENUM // Cardiologist | Dermatologist | ...
licenseNumber int
available boolean
patientCount int // cap: 5
hospitalName String
version int // optimistic lock
appointment-service :8084

Orchestrates the booking lifecycle. Validates patient and doctor IDs via REST before persisting. On payment confirmation, publishes to Kafka. A scheduled job runs daily at 06:00 to delete expired paid appointments using a single bulk-delete query — no records loaded into JVM memory.

patientId UUID // FK ref, not enforced at DB level
doctorId UUID
serviceDate String // "yyyy-MM-dd HH:mm"
serviceType ENUM // CONSULTATION | SURGERY | VACCINATION
amount float
paymentStatus boolean
paymentType ENUM // CREDIT | DEBIT
auth-service :8089

Issues JWTs signed with HMAC-SHA256. The same signing key is shared with the API Gateway via environment variable APP_SECRET. Passwords are hashed with BCrypt. Sessions are fully stateless.

id UUID // GenerationType.UUID
name String // max 50 chars, unique
email String // validated @Email
password String // BCrypt hash
registerDate LocalDate

Request Flow — Authentication

Auth endpoints (/api/auth/**) are whitelisted at the gateway — no JWT filter runs. The gateway strips the /api prefix and forwards to auth-service on port 8089. The auth-service itself has its own Spring Security config that permits /auth/** without a session.

JWT Verification at Gateway

For all non-auth routes, the gateway's jwtAuthenticationFilter (WebFilter) extracts the Bearer token, parses claims using Keys.hmacShaKeyFor(APP_SECRET), extracts roles into SimpleGrantedAuthority objects, and writes the authentication into ReactiveSecurityContextHolder. Clock skew tolerance is set to 5 minutes. Downstream services receive the request without further auth checks.

04

Request Flow — Appointment Creation

Appointment creation is the most complex write path in the system. It involves synchronous cross-service validation with circuit-breaker protection, a Kafka publish, and an optional scheduled cleanup job.

01
Inbound — Gateway routes POST /api/appointments/

JWT validated at gateway. StripPrefix=1 applied. Request lands at AppointmentController.createAppointment() as raw Appointment entity, mapped to CreateAppointmentServiceRequestDto by AppointmentMapper.

02
Patient ID validation — Resilience4j + Retry

IdValidation.checkPatientExists(UUID) calls GET patient-management:8080/patients/find/{id} via RestTemplate. Annotated with @CircuitBreaker(name="patientService", fallbackMethod="patientFallback") and @Retry(name="patientService"). Config: sliding-window=10, failure-rate-threshold=50%, wait-in-open=10s, max-attempts=3, wait=500ms. On fallback, returns false → CustomNotFoundException thrown.

03
Doctor ID validation — same pattern

Identical circuit-breaker configuration against doctor-service:8083/doctors/find/{id}. Both calls are sequential, not concurrent.

04
Persist to PostgreSQL

appointmentRepository.save(appointment) via Spring Data JPA. HikariCP pool: max=10, min-idle=5, connection-timeout=30s.

05
Payment confirmation — Kafka publish

When PATCH /appointments/appointment-update/{id}/{status} sets paymentStatus=true, KafkaProducer.sendEvent() serializes an AppointmentKafkaResponseDto to JSON via Jackson and sends to topic appointment-payment-updated. Billing service consumes this event.

06
Scheduled cleanup — AppointmentCleanupService

Runs at cron = "0 0 6 * * ?" (daily 06:00). Executes a single JPQL bulk-delete: DELETE FROM Appointment WHERE paymentStatus=true AND serviceDateEnd < :cutoff. No entities loaded into memory. Returns delete count for logging.

05

Request Flow — Automated Invoicing

The billing service has no inbound REST endpoints for invoice creation — it is purely event-driven. KafkaConsumer listens on appointment-payment-updated with consumer group appointment-group.

01
Consume — deserialize payload

KafkaConsumer.listen(String message) deserializes to AppointmentDTO via ObjectMapper.readValue(). Contains doctorId, patientId, amount, paymentStatus.

02
Build invoice ID

Invoice number: patientId + "-" + doctorId + "-" + System.currentTimeMillis(). Guarantees uniqueness without a database sequence.

03
Call external PDF API

InvoiceService.generateInvoice() builds a JSON body with line items and POSTs to invoice-generator.com via RestTemplate with Authorization: Bearer {INVOICE_API_KEY}. Response is byte[].

04
Write PDF to disk

Files.write(invoicePath, response.getBody()) writes to ./invoices/{invoiceNumber}.pdf. Directory created if absent via Files.createDirectories().

Known Limitation

Invoice PDFs are written to the container's local filesystem. In a multi-replica deployment this would require a shared volume or object storage (S3). Current setup is single-instance per service.

Request Flow — API Gateway Routing

The gateway is built on Spring Cloud Gateway (WebFlux / Project Reactor). All route definitions live in application.yml. There is no service discovery — routes are statically configured to Docker Compose service names.

Inbound Path Upstream Filters
/api/patients/** patient-management:8080 StripPrefix=1
/api/doctors/** doctor-service:8083 StripPrefix=1
/api/appointments/** appointment-service:8084 StripPrefix=1
/api/auth/** auth-service:8089 StripPrefix=1, SetRequestHeader Content-Type
/api-docs/patients patient-management:8080 RewritePath → /v3/api-docs
/auth/**, /actuator/** — Permit all — no JWT filter
WebFlux Security Note

The gateway uses @EnableWebFluxSecurity, not standard MVC security. ReactiveUserDetailsService is overridden with a no-op bean to suppress Spring Security's autoconfigured form login. Auth is handled entirely by the custom jwtAuthenticationFilter WebFilter.

07

Fault Tolerance — Resilience4j

Circuit breakers guard all outbound REST calls from appointment-service to patient-service and doctor-service. Configuration is identical for both:

# appointment-service / application.properties

resilience4j.circuitbreaker.instances.patientService
  sliding-window-size                    = 10
  failure-rate-threshold                 = 50      # % failures to open
  wait-duration-in-open-state            = 10s
  permitted-calls-in-half-open-state     = 3
  minimum-number-of-calls               = 5

resilience4j.retry.instances.patientService
  max-attempts                           = 3
  wait-duration                          = 500ms

State transitions: CLOSED → (failure rate exceeds threshold) → OPEN → (wait 10s) → HALF_OPEN → (3 probe calls pass) → CLOSED. In OPEN state, all calls immediately hit the fallback method which returns false, causing the appointment creation to fail with CustomNotFoundException rather than timing out.

patient-service also configures a circuit breaker around its Kafka producer to prevent Kafka broker unavailability from blocking HTTP request threads.

Every service exposes a /actuator/prometheus endpoint via Micrometer. Prometheus scrapes all 6 services on a 5-second interval. Grafana reads from Prometheus and serves a pre-provisioned dashboard.

Load testing

Three k6 scripts cover different load profiles: low-stress.js (10 VUs, 30s), medium-stress.js (ramp to 50 VUs), intense-stress.js (ramp to 200 VUs). appointments-stress.js runs a full setup phase — registers a user, creates 50 patients and 50 doctors, then hammers appointment creation at 100 req/s constant arrival rate for 30 seconds. Thresholds: P95 < 2000ms, error rate < 1%.

Test Coverage

patient-service achieves 72% instruction coverage measured by JaCoCo. Coverage spans controller (@WebMvcTest), service (Mockito unit tests), repository (@DataJpaTest with H2), and Kafka producer. DTOs, entity classes, config, exception handlers, and generated Protobuf classes are excluded from the report.

CI / CD

Each service has its own GitHub Actions workflow triggered on push to master when files under its directory change. This prevents unrelated service rebuilds.

# Per-service workflow pattern
on:
  push:
    branches: [master]
    paths:
      - 'appointment-service/**'

# Kubernetes workflow uses matrix strategy across 6 services
strategy:
  matrix:
    service:
      - { name: api-gateway,          port: 4004 }
      - { name: appointment-service,   port: 8084 }
      - ...

The kubernetes-register.yml workflow builds multi-platform images (linux/amd64,linux/arm64) using Docker Buildx, pushes to GHCR, runs Trivy vulnerability scanning, and updates image tags in Kubernetes manifests. Dockerfile uses a two-stage build: Maven builder image + slim eclipse-temurin:21-jdk runner. JVM flags: -XX:MaxRAMPercentage=75.0 -XX:+UseG1GC -XX:+ExitOnOutOfMemoryError.
